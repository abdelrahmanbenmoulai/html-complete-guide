< Introduction to HTML />

What is HTML?

HTML (HyperText Markup Language) is the standard markup language for creating web pages. It describes the structure of a web page and consists of a series of elements that tell the browser how to display the content.

Evolution of HTML

HTML 1.0

Created by Tim Berners-Lee in 1991

HTML 2.0

Released in 1995

HTML 3.2

Released in 1997

HTML 4.01

Released in 1999

XHTML

Released in 2000

HTML5

Released in 2014 - Current standard

HTML5 introduced many new features including semantic elements, form improvements, multimedia support, and better mobile compatibility.

How HTML Works

HTML uses markup tags to describe different parts of a webpage:

Basic HTML Document 
<!DOCTYPE html>
        <html>
        <head>
            <title>Page Title</title>
        </head>
        <body>
            <h1>My First Heading</h1>
            <p>My first paragraph.</p>
        </body>
        </html>
DOCTYPE
HTML Tag
HEAD Tag
BODY Tag

The <!DOCTYPE html> declaration defines the document type and version of HTML. It helps browsers render the page correctly.

The <html> element is the root element of an HTML page that contains all other elements.

The <head> element contains meta information about the document, such as the title, character set, styles, and scripts.

The <body> element contains the visible content of the web page, including text, images, links, and other elements.

Key Characteristics of HTML

File Extension

HTML documents have a .html or .htm file extension

Case Sensitivity

HTML is not case-sensitive, but best practice is to use lowercase

Browser Interpretation

HTML documents are interpreted by web browsers

Web Foundation

HTML is the building block of the web, often used with CSS and JavaScript

While HTML provides the structure of a webpage, CSS is used for styling, and JavaScript adds interactivity. Together, they form the foundation of modern web development.

What are HTML Elements?

HTML elements are the building blocks of HTML pages. Elements are represented by tags, usually coming in pairs:

Basic Element Syntax 
<tagname>Content goes here...</tagname>

Anatomy of an HTML Element

Opening Tag

Marks the beginning of an element (<tagname>)

Content

The information contained between tags

Closing Tag

Marks the end of an element (</tagname>)

Together, these three parts form a complete HTML element. The opening and closing tags act like containers that tell the browser what content they enclose.

Nested HTML Elements

HTML elements can be nested (elements can contain elements):

Nested Elements Example 
<html>

            <h1>This is a heading</h1>
            <p>This is a <strong>paragraph</strong>.</p>
          </body>
        </html>

Importance of Headings

SEO

Search engines use headings to index the structure and content of your web pages

Scannability

Headings help visitors quickly scan your pages and understand the content structure

Accessibility

Headings are important for accessibility (screen readers use them to navigate)

Best Practices for Headings

  • Use headings for structure, not for making text bold or big
  • Maintain a proper hierarchy (don't skip heading levels, like from <h1> to <h3>)
  • Each page should have only one <h1> heading
  • Keep headings concise and descriptive
  • Use CSS to change the appearance of headings if needed
Good Structure
Bad Structure
Good Heading Structure 
<h1>Website Title</h1>
<h2>Main Section</h2>
<p>Some content here...</p>
<h3>Subsection</h3>
<p>More content here...</p>
<h3>Another Subsection</h3>
<p>Additional content...</p>
<h2>Another Main Section</h2>

This structure follows a clear hierarchy and makes logical sense for both users and search engines.

<h1>Main Section</h1>

<h4>Subsection</h4>

<h2>Another Subsection</h2>
<p>Additional content...</p>

This structure is confusing because it doesn't follow a logical hierarchy. It starts with <h3>, jumps to <h1>, then to <h4>, and then to <h2>.

Styling Headings

You can style headings with CSS:

Inline CSS for Headings

Or in an external or internal CSS:

CSS for Headings

    font-size: 36px;
    color: blue;
    text-align: center;
}

This is a styled heading

While you can style headings to look any way you want, remember that their primary purpose is to create a logical document structure. The visual appearance should support this structure, not replace it.

Semantic Value of Headings

Headings aren't just about size and appearance. They carry semantic meaning that helps define the structure of your content. This semantic structure is important for:

SEO Ranking

When using text formatting elements, keep accessibility in mind:

Accessibility Best Practices for Links

  • Use descriptive link text that makes sense out of context (avoid "click here" or "read more")
  • Add aria-label to links that use only icons or images
  • Ensure sufficient color contrast between link text and background
  • Make the clickable area large enough for easy interaction (especially on mobile)
  • Use title attributes to provide additional context when needed
  • Indicate when links open in a new window or download files

URL Encoding

When adding parameters to URLs, special characters need to be encoded. For example, spaces become %20, and other special characters have their own encodings. Most common encodings:

  • Space: %20
  • ?: %3F
  • &: %26
  • =: %3D
  • /: %2F

You can use JavaScript's encodeURIComponent() function to properly encode parameter values.

JavaScript Link Best Practices

When using JavaScript with links:

Security Concerns

Iframes can pose security risks. Use the sandbox attribute to restrict capabilities:

Sandboxed Iframe 
<iframe src="https://example.com" sandbox="allow-scripts allow-same-origin"></iframe>
Sandbox Values
Feature Policies
Security Tips

Common sandbox values:

  • allow-scripts - Allows scripts to run
  • allow-same-origin - Allows content to be treated as same origin
  • allow-forms - Allows form submission
  • allow-popups - Allows popups from iframe
  • allow-top-navigation - Allows iframe to navigate its top-level browsing context
  • allow-modals - Allows the iframe to open modal windows

Using sandbox without any values applies all restrictions. Add specific permissions only as needed.

Control which features are available to the iframe using the allow attribute:

Feature Policy Example 
<iframe src="https://example.com" allow="camera; microphone"></iframe>

Common feature policy values:

  • camera - Access to the camera
  • microphone - Access to the microphone
  • geolocation - Access to location data
  • autoplay - Ability to autoplay media
  • fullscreen - Ability to enter fullscreen mode
  • payment - Access to the Payment Request API

Best practices for secure iframe usage:

  • Always use the sandbox attribute with minimal permissions
  • Set appropriate allow directives to limit feature access
  • Consider using Content-Security-Policy headers
  • Only embed content from trusted sources
  • Use HTTPS for both your page and embedded content
  • Set proper X-Frame-Options headers on your own content

Be aware of clickjacking attacks where malicious sites embed your content in iframes to trick users.

Async and Defer

The async and defer attributes control how external scripts are loaded and executed:

async

<script src="script1.js" async></script>

Downloads asynchronously, executes as soon as possible

  • Doesn't block HTML parsing while downloading
  • Executes immediately when downloaded
  • May execute before the DOM is fully loaded
  • Order of execution not guaranteed

defer

<script src="script2.js" defer></script>

Downloads asynchronously, executes after parsing

  • Doesn't block HTML parsing while downloading
  • Executes after HTML parsing is complete
  • Executes before the DOMContentLoaded event
  • Maintains order of execution

Best Practice: Use defer for scripts that need the DOM or depend on other scripts executing in a specific order. Use async for independent scripts like analytics.

Script Loading Comparison 
<!-- Regular: Blocks parsing, executes immediately -->
<script src="regular.js"></script>

<!-- Async: Non-blocking download, executes ASAP -->
<script src="analytics.js" async></script>

<!-- Defer: Non-blocking download, executes after parsing -->
<script src="app.js" defer></script>

< HTML File Paths />

File paths tell the browser where to find resources like images, stylesheets, scripts, and linked pages. Understanding how to correctly specify file paths is essential for creating functional websites.

File Path Types

Absolute File Paths
Relative File Paths

Absolute file paths point to a location defined from the root directory or include the full URL. They work regardless of where the current HTML file is located.

Absolute File Path Examples 
<!-- Full URL -->
      <img src="https://www.example.com/images/picture.jpg" alt="Picture">
      <a href="https://www.example.com/about.html">About</a>
      
      <!-- From root directory of the current domain -->
      <img src="/images/picture.jpg" alt="Picture">
      <a href="/about.html">About</a>

Paths starting with a forward slash / (without http:// or https://) point to the root directory of the current domain.

Advantages

  • Works from any page on your site
  • Remains valid if the current page moves
  • Required for external resources

Disadvantages

  • Breaks if domain or folder structure changes
  • Not suitable for development environments that might use different domains
  • Longer to write than relative paths

Relative file paths point to a location relative to the current HTML file. They're useful for resources within your own website.

Relative File Path Examples 
<!-- Same directory -->
      <img src="picture.jpg" alt="Picture">
      <a href="about.html">About</a>
      
      <!-- Subdirectory -->
      <img src="images/picture.jpg" alt="Picture">
      <a href="pages/about.html">About</a>
      
      <!-- Parent directory -->
      <img src="../picture.jpg" alt="Picture">
      <a href="../about.html">About</a>
      
      <!-- Two levels up -->
      <img src="../../picture.jpg" alt="Picture">

../ means "go up one directory level." You can chain these to go up multiple levels.

Advantages

  • Shorter and cleaner syntax
  • Works when moving entire projects
  • Works in local development environments

Disadvantages

  • Breaks if the current file moves
  • Can become complex with deeply nested directories
  • Requires understanding of directory structure

Consistent Structure

Organize your website with a clear, logical folder structure. Common folders include:

  • css/ for stylesheets

  • pages/ for additional HTML pages

File Naming

  • Use hyphens instead of spaces in filenames
  • Be consistent with file extensions
  • Use descriptive, but concise names

Path Selection

  • Use relative paths for files within your own website
  • Use absolute paths for external resources

Testing

  • Verify paths work when deployed to a server

Base Element

The <base> element specifies a base URL for all relative URLs in a document:

Base Element Example 
<head>
        <base href="https://www.example.com/pages/">
        <!-- Now all relative URLs will be relative to https://www.example.com/pages/ -->
      </head>
      <body>
      <!-- This will link to https://www.example.com/pages/about.html -->
      <a href="about.html">About</a>
      
      <!-- This will look for the image at https://www.example.com/pages/images/logo.png -->
      <img src="images/logo.png" alt="Logo">
    </body>

The <base> element must be inside the <head> section, and there can only be one <base> element in a document. It's particularly useful for single-page applications or when moving pages between different environments.

URL Encoding

When file paths contain special characters or spaces, they need to be URL encoded:

URL Encoding Example 
<!-- Incorrect (contains space) -->
    <img src="images/product image.jpg" alt="Product">
    
    <!-- Correct (URL encoded) -->
    <img src="images/product%20image.jpg" alt="Product">

Common URL encodings:

  • Space: %20
  • ?: %3F
  • &: %26
  • =: %3D
  • #: %23

Best practice: Avoid special characters in filenames altogether.

The <meta> element provides metadata about the HTML document. Metadata is information about data that isn't displayed on the page but is machine parsable.

Character Encoding

<meta charset="UTF-8">

Specifies the character encoding for the HTML document. UTF-8 is the recommended encoding as it covers almost all characters and symbols in the world.

Viewport Settings

<meta name="viewport" content="width=device-width, initial-scale=1.0">

Essential for responsive design. This tag tells the browser how to control the page's dimensions and scaling on different devices.

<meta name="description" content="This is my website about web development">

Provides a brief description of the page. Search engines often display this in search results. Aim for 150-160 characters.

Keywords

<meta name="author" content="John Doe">

Preloading and Resource Hints

Modern browsers support various ways to optimize resource loading through the head section.

Resource Hints 
<!-- Preconnect to important third-party domains -->
    <link rel="preconnect" href="https://fonts.googleapis.com">
    <link rel="preconnect" href="https://analytics.example.com" crossorigin>
    
    <!-- Prefetch resources you'll need for the next page -->
    <link rel="prefetch" href="next-page.html">
    
    <!-- Preload critical resources for current page -->
    <link rel="preload" href="critical-style.css" as="style">
    <link rel="preload" href="hero-image.jpg" as="image">
    <link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin>
    
    <!-- DNS prefetching (older technique) -->
    <link rel="dns-prefetch" href="https://api.example.com">

Preconnect

Informs the browser to establish early connections to important third-party domains. This includes DNS lookups, TLS negotiations, and TCP handshakes.

Prefetch

Tells the browser to download resources that will be needed for the next navigation. These are fetched with low priority during idle time.

Preload

Instructs the browser to download key resources needed for the current page as soon as possible. These are high-priority fetches.

DNS Prefetch

An older technique that only performs DNS lookups. Less powerful than preconnect but has broader browser support.

Controls how much referrer information should be included with requests. Important for privacy and security.

< HTML URL Encoding />

URL encoding is a mechanism for translating unprintable or special characters to a format that can be transmitted over the Internet. It ensures that URLs remain valid and functional across different systems and browsers.

What is URL Encoding?

URL encoding replaces unsafe ASCII characters with a "%" followed by hexadecimal digits. URLs can only use a specific subset of ASCII characters, so any characters outside this set must be encoded.

For example, a space character is encoded as "%20" or "+" (in query strings), and the ampersand "&" is encoded as "%26".

When to Use URL Encoding

Unsafe Characters

URLs cannot contain spaces, control characters, and certain special characters that have specific meanings in URL syntax. 

<!-- Incorrect -->
    <a href="search.php?query=HTML & CSS">Search</a>
    
    <!-- Correct -->
    <a href="search.php?query=HTML%20%26%20CSS">Search</a>

Non-ASCII Characters

Characters outside the ASCII set (like accented letters, non-Latin scripts) need to be encoded to ensure proper transmission. 

<!-- Encoding non-ASCII characters -->
    <a href="product.php?name=%C3%BCber">Über</a>

Reserved Characters

Some ASCII characters are reserved for URL syntax and must be encoded when used as part of the data.

  • Reserved characters: : / ? # [ ] @ ! $ & ' ( ) * + , ; =
  • Unreserved characters (safe to use): a-z A-Z 0-9 - _ . ~

Percent Character

The percent sign (%) is used to introduce encoded characters, so it must itself be encoded as %25 when used as data. 

<!-- Encoding a percent sign -->
    <a href="discount.php?rate=25%25">25% Off</a>

< HTML Versus XHTML />

HTML and XHTML are markup languages used for creating web pages, but they differ in their syntax rules and strictness. Understanding these differences helps you write more consistent and compatible code.

What is XHTML?

XHTML (eXtensible HyperText Markup Language) is a stricter, more XML-based version of HTML. It was developed to make HTML more extensible and to improve interoperability with other data formats.

XHTML combines the strength of HTML with the extensibility of XML, requiring web developers to follow more rigorous coding standards.

Key Differences

Document Declaration
Syntax Rules
Elements & Attributes

The DOCTYPE declaration differs between HTML5 and XHTML: 

<!-- HTML5 DOCTYPE (simple) -->
    <!DOCTYPE html>

    <!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Strict//EN"
    "http://www.w3.org/TR/xhtml1/DTD/xhtml1-strict.dtd">
    
    <!-- XHTML 1.1 DOCTYPE -->
    <!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.1//EN"
    "http://www.w3.org/TR/xhtml11/DTD/xhtml11.dtd">

HTML5's DOCTYPE is much simpler, reflecting its more flexible approach compared to XHTML's strict XML-based requirements.

XHTML enforces stricter syntax rules than HTML:

Case Sensitivity

HTML: Case-insensitive (can use uppercase, lowercase, or mixed)

XHTML: Case-sensitive and requires lowercase element and attribute names

Element Closing

HTML: Some elements can be left unclosed (like <br>, <img>)

XHTML: All elements must be closed, either with a closing tag or self-closing syntax

Attribute Values

HTML: Quotes around attribute values are optional in some cases

XHTML: All attribute values must be enclosed in quotes

Nesting

HTML: Browsers may forgive improper nesting

XHTML: Elements must be properly nested

Key XHTML requirements for elements and attributes:

  • All element names must be in lowercase
  • All elements must be closed
  • Empty elements must be closed with a trailing slash (e.g., <br />)
  • All attribute names must be in lowercase
  • All attribute values must be quoted
  • Minimized attributes (like "checked") must have a value (e.g., checked="checked")

HTML and XHTML handle errors differently:

HTML Error Handling

HTML is more forgiving. Browsers will attempt to render the page even with syntax errors, often making "best guesses" about the intended structure.

This can lead to unexpected rendering but rarely prevents the page from displaying.

XHTML Error Handling

XHTML is parsed as XML, which means a single syntax error can prevent the entire page from rendering.

When served with the correct MIME type (application/xhtml+xml), browsers will show an error message instead of attempting to render a page with XML syntax errors.

Note: Most XHTML pages on the web are actually served with a text/html MIME type, which causes browsers to parse them as HTML, negating many of the strict XML benefits and requirements.

The <track> element enhances media accessibility by adding text tracks to audio and video content. These tracks can provide subtitles, captions, descriptions, and more, making your media content accessible to a wider audience.

Basic <track> Element Usage

The <track> element is always used as a child of an <audio> or <video> element. It allows you to specify timed text tracks for media playback.

Track files are typically in WebVTT (.vtt) format, a standard format for displaying timed text tracks.

<track> Element Attributes

src

Specifies the URL of the track file. 

<track src="captions.vtt">

This is a required attribute that points to a text track file, typically in WebVTT (.vtt) format.

kind

Specifies the type of text track. 

<track kind="subtitles">

Possible values:

  • subtitles - Translations of foreign language content
  • captions - Transcription of audio, may include non-speech elements
  • descriptions - Text descriptions of video content
  • chapters - Chapter titles for navigation
  • metadata - Tracks used by scripts, not visible to users

srclang

Specifies the language of the track text. 

<track srclang="en">

This attribute uses language codes (e.g., "en" for English, "es" for Spanish). It's required if kind="subtitles".

label

Specifies a user-readable title for the track. 

<track label="English Subtitles">

This title appears in the browser's interface when users select from available text tracks.

default

Indicates that this track should be enabled by default. 

<track default>

Only one track should have the default attribute. If no track has this attribute, none will be enabled by default.

WebVTT (Web Video Text Tracks) is the standard format for <track> elements. Here's a basic example:

      00:00:10.000 --> 00:00:14.000

WebVTT File Structure:

  1. The file must start with "WEBVTT" on the first line
  2. Each caption is separated by a blank line
  3. Each caption has a timestamp range in the format "HH:MM:SS.mmm --> HH:MM:SS.mmm"
  4. The text for the caption follows the timestamp

function applyFilter(canvas, filter) {
  var ctx = canvas.getContext("2d");
  var imageData = ctx.getImageData(0, 0, canvas.width, canvas.height);
  var data = imageData.data;
  
  switch(filter) {
    case "grayscale":
      for (var i = 0; i < data.length; i += 4) {
        var avg = (data[i] + data[i + 1] + data[i + 2]) / 3;
        data[i]     = avg; // red
        data[i + 1] = avg; // green
        data[i + 2] = avg; // blue
      }
      break;
    case "invert":
      for (var i = 0; i < data.length; i += 4) {
        data[i]     = 255 - data[i];     // red
        data[i + 1] = 255 - data[i + 1]; // green
        data[i + 2] = 255 - data[i + 2]; // blue
      }
      break;
    case "sepia":
      for (var i = 0; i < data.length; i += 4) {
        var r = data[i];
        var g = data[i + 1];
        var b = data[i + 2];
        
        data[i]     = Math.min(255, (r * 0.393) + (g * 0.769) + (b * 0.189));
        data[i + 1] = Math.min(255, (r * 0.349) + (g * 0.686) + (b * 0.168));
        data[i + 2] = Math.min(255, (r * 0.272) + (g * 0.534) + (b * 0.131));
      }
      break;
    case "reset":
      // This assumes we have the original image stored
      drawImageToCanvas();
      return;
  }
  
  ctx.putImageData(imageData, 0, 0);
}

Canvas Performance Tips

Optimize Canvas Performance:

  1. Reduce canvas size - Use the smallest canvas dimensions that meet your needs
  2. Avoid unnecessary drawing - Only redraw what changes
  3. Use multiple canvas layers - Separate static and dynamic content
  4. Batch drawing operations - Minimize context state changes
  5. Use requestAnimationFrame - For smoother animations
  6. Avoid shadows and complex gradients - They're expensive to render
  7. Use canvas scaling - Draw on a smaller canvas and scale up if appropriate
  8. Avoid floating-point coordinates - Use integers when possible
  9. Disable transparency - Use canvas.getContext('2d', { alpha: false }) when you don't need transparency
  10. Consider using WebGL - For complex graphics that can benefit from GPU acceleration

  • For interactive canvas elements, use ARIA roles and properties
  • Consider providing an alternative representation (table, list, etc.) for data visualizations
  • Ensure keyboard navigation for interactive canvas applications

What is SVG?

SVG (Scalable Vector Graphics) is an XML-based vector image format for two-dimensional graphics. Unlike raster formats (like JPEG, PNG), SVG defines graphics in terms of shapes, paths, text, and filter effects that can scale to any size without losing quality.

As an XML-based format, SVG elements become part of the HTML DOM, making them accessible and manipulable through CSS and JavaScript.

      What is Responsive Web Design?
      
        
      
          Responsive Web Design makes web pages look good on all devices by automatically adjusting for different screen
          sizes and viewports.
        
      

        
      Key Components
      
        
      
          
      
            
      
              
            
      
            
      Viewport Meta Tag
      
            
      Essential for controlling page dimensions on different devices
      
          
      
          
      
            
      
              
            
      
            
      Responsive Images
      
            
      Images that scale well on different screen sizes
      
          
      
          
      
            
      
              
            
      
            
      Responsive Text
      
            
      Text that adjusts based on viewport size
      
          
      
          
      
            
      
              
            
      
            
      Media Queries
      
            
      Apply different styles based on device characteristics
      
          
      
          
      
            
      
              
            
      
            
      Layout Frameworks
      
            
      CSS frameworks that help create responsive designs
      
          
      
        
      

        
      Viewport Meta Tag
      
        
      
          The viewport meta tag is essential for responsive designs as it instructs the browser how to control the
          page's dimensions and scaling.
        
      

        
      
          
      
            Viewport Meta Tag
            
          
      
          
      <meta name="viewport" content="width=device-width, initial-scale=1.0">
      
        
      

        
      Responsive Images
      
        
      Images that scale well on different screens are crucial for responsive design:
      

        
      
          
      
            Basic Responsive Image
            
          
      
          
      <img src="image.jpg" style="max-width:100%;height:auto;" alt="Responsive Image">
      
        
      

        
      
          
      
            Picture Element for Art Direction
            
          
      
          
      <picture>
  <source media="(min-width: 768px)" srcset="large.jpg">
  <source media="(min-width: 480px)" srcset="medium.jpg">
  <img src="small.jpg" alt="Responsive Image" style="width:auto;">
</picture>
      
        
      

        
      Responsive Text
      
        
      Using viewport-relative units helps create text that scales with the screen size:
      

        
      
          
      
            Viewport-Relative Text
            
          
      
          
      <h1 style="font-size:5vw;">Responsive Text</h1>
      
        
      

        
      Media Queries
      
        
      Media queries allow you to apply different styles based on device characteristics:
      

        
      
          
      
            Basic Media Queries
            
          
      
          
      /* For desktop */
.container {
  width: 960px;
}

/* For tablets */
@mediaonlyscreenand(max-width: 768px) {
  .container {
    width: 720px;
  }
}

/* For mobile phones */
@mediaonlyscreenand(max-width: 480px) {
  .container {
    width: 100%;
  }
}

      CSS frameworks like Bootstrap make it easier to create responsive layouts:

      <div class="container">
  <div class="row">
    <div class="col-sm-6 col-md-4 col-lg-3">Column 1</div>
    <div class="col-sm-6 col-md-4 col-lg-3">Column 2</div>
    <div class="col-sm-6 col-md-4 col-lg-3">Column 3</div>
    <div class="col-sm-6 col-md-4 col-lg-3">Column 4</div>
  </div>
</div>
      
        
      

        
      Responsive Design Best Practices
      
        
      
          
      
            
      Mobile First
      
            
      Relative Units
      
            
      Testing
      
            
      Optimization
      
          
      
          
      
            
      
              
      
                Design for mobile devices first, then enhance the experience for larger screens. This approach ensures
                your site works well on smaller devices.
              
      
            
      
            
      
              
      
                Use relative units (%, vw, vh, em, rem) instead of fixed pixels to ensure elements scale proportionally
                across different screen sizes.
              
      
            
      
            
      
              
      
                Test your responsive design on multiple devices and browsers to ensure consistent user experience. Use
                browser developer tools to simulate different screen sizes.
              
      
            
      
            
      
              
      
                Optimize images for different screen sizes using responsive images techniques. Consider touch targets
                (make buttons and links large enough for touch interaction).
              
      
            
      
          
      
        
      

        
      
          
      
          
      
            
      
              Always include the viewport meta tag in your HTML documents to ensure proper rendering on mobile devices.
              This is the foundation of any responsive design.
            
      
          
      
        
      

        
      
          
      
          
      
            
      
              Common breakpoints for media queries are 480px (mobile), 768px (tablet), 992px (desktop), and 1200px
              (large desktop), but you should adjust these based on your specific design needs.
